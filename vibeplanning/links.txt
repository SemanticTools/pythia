PYTHIA LINK MODE — IMPLEMENTATION PLAN
========================================
Date: 2026-03-01

OVERVIEW
--------
"Link Mode" is a keyboard-driven overlay for storing, finding, and launching URLs.
Activate it from anywhere in the app with a single keystroke. Inside link mode,
single-key commands and instant fuzzy search let you get to any link in 2-3
keystrokes. Escape or the same toggle key exits.

URLs pasted into the main ask textarea are saved to links automatically and
silently — no confirmation, no extra UI.

KEYSTROKES INSIDE LINK MODE
  ↑ / ↓         move selection up/down
  Enter         open selected link in new browser tab
  Type text     instant fuzzy filter (updates list as you type)
  #<text>       tag-only filter: type # then partial tag name (see TAG SEARCH)
  Ctrl+V        paste — if clipboard is a URL, jumps to "add" flow
  a             open add-link inline form (pre-fills URL from clipboard if valid)
  d             delete selected link (confirm with y/n prompt in the status bar)
  e             edit selected link inline (title, tags, note)
  t             quick tag editor on selected link (see TAG EDITING)
  c             copy selected URL to clipboard
  /             focus search box (in case it lost focus)
  Escape        exit link mode (or cancel current inline form)
  ?             show link mode help overlay

PASTE / ADD WORKFLOW (via link mode)
  1. User has a URL in clipboard.
  2. Press Ctrl+L to enter link mode.
  3. Press Ctrl+V (or 'a') — URL is pre-filled in the add form.
  4. Backend fetches the page <title> automatically.
  5. User can accept title, add optional tags, press Enter to save.
  6. Link appears at top of list. Total keystrokes: ~5.

PASTE INTO MAIN ASK PROMPT (silent auto-save)
  When a URL is pasted into the #question textarea, it is saved to links
  automatically with no UI and no interruption to the ask flow.

  1. User pastes a URL into the ask textarea.
  2. URL is saved silently: POST /links { url }.
  3. Title is fetched in the background and backfilled via PATCH /links/:id.
  4. The textarea is unaffected — user continues writing their question.

  Only clean URL pastes trigger this (entire pasted string must be a URL).
  Mixed text pastes are ignored.

QUICK FIND + LAUNCH WORKFLOW
  1. Press Ctrl+L.
  2. Type a few chars of title, URL, or tag.
  3. ↓ to select if needed.
  4. Enter to open.
  Total keystrokes from any state: ~4-6.

SEARCH BEHAVIOR
  - Instant client-side filter (no server round-trip).
  - Searches: title, URL, tags, note.
  - Tags are first-class: partial tag matches rank above title/URL matches.
  - Most-visited links float up when search box is empty.
  - Visits counter incremented on each open.
  - See FUZZY SEARCH ALGORITHM for full scoring.

TAG SEARCH (# prefix)
  Typing # in the search box switches to tag-only filter mode.
  Only links where at least one tag starts with (or contains) the typed
  partial are shown. Everything else is hidden.

  Examples:
    #prod   → shows links tagged "prod", "production", "prod-db"
    #dep    → shows links tagged "deploy", "deployment", "devops"
    #       → shows all links that have any tag (useful for browsing)

  Visual cue: search box border highlights (accent color) when in tag mode.
  Normal typing (no #) returns to full search immediately.

  After filtering, ↑/↓/Enter work exactly the same — walk to the link,
  press Enter to launch.

TAG EDITING ('t' key — quick tag editor)
  Pressing 't' on a selected link opens an inline tag editor in #link-status
  without leaving link mode. The workflow:

    ┌──────────────────────────────────────────────────────┐
    │ Tags: [prod] [deploy] ×    [____________] ↓          │
    │        ↑ existing chips     ↑ tag input  autocomplete│
    └──────────────────────────────────────────────────────┘

  - Existing tags shown as removable chips (× to remove).
  - Type in the input: a dropdown of matching existing tags appears below.
  - ↓ / Tab  move into the autocomplete dropdown.
  - Enter    select highlighted suggestion OR commit typed tag if no dropdown.
  - Space or comma  commit the current typed tag (add it, clear input).
  - Backspace on empty input  removes the last tag chip.
  - Enter on empty input  saves and closes the tag editor.
  - Escape  cancels without saving.

  Tag autocomplete source: all unique tags across all links in _links,
  filtered to those starting with the current input (case-insensitive).
  Sorted alphabetically. Max 6 suggestions shown.

STORAGE FILE:  data/links.json
FORMAT:        JSON array, sorted by addedAt descending on write.

LINK OBJECT:
  {
    "id":      "uuid-v4",
    "url":     "https://...",
    "title":   "Page Title",         // auto-fetched, editable
    "tags":    ["tag1", "tag2"],     // optional, array of strings
    "note":    "short free text",    // optional
    "addedAt": "2026-03-01T12:00Z",
    "visits":  0                     // incremented each time link is opened
  }

No Qdrant needed — all links are kept in memory at runtime (links.json is
small, typically <1000 entries). Fuzzy search runs in JS on the full array.

NEW FILE:  src/links.mjs
  - loadLinks()          read data/links.json, return array (empty if missing)
  - saveLinks(arr)       write data/links.json
  - addLink({url,title,tags,note})   create, assign uuid, addedAt, visits:0
  - updateLink(id, patch)            merge patch into existing record
  - deleteLink(id)                   remove by id
  - bumpVisit(id)                    increment visits

NEW ENDPOINTS in server.mjs:

  GET    /links            → 200 { links: [...] }
  POST   /links            → 201 { link: {...} }
                             body: { url, title?, tags?, note? }
  PATCH  /links/:id        → 200 { link: {...} }
                             body: any subset of { url, title, tags, note }
  DELETE /links/:id        → 204
  POST   /links/:id/visit  → 204   (increments visits counter)
  GET    /links/fetch-title?url=...  → 200 { title: "..." }
                             server fetches the URL, parses <title> tag,
                             returns it. Times out at 5s. Returns "" on fail.

FUZZY SEARCH ALGORITHM (client-side, no library):

  If query starts with '#':
    tagQuery = query.slice(1).toLowerCase()
    For each link: include if any tag starts with tagQuery (or contains it).
    Score = length of longest matching tag prefix (longer prefix = better match).
    No title/URL/note scoring in tag mode.
    Special case: '#' alone (empty tagQuery) → show all links that have tags.

  Otherwise (normal mode):
    q = query.toLowerCase()
    For each link, compute score:
      +40  any tag starts with q  (partial tag prefix — highest non-title priority)
      +35  any tag === q          (exact tag match)
      +30  title.toLowerCase().startsWith(q)
      +20  title.toLowerCase().includes(q)
      +10  url.toLowerCase().includes(q)
      + 5  note.toLowerCase().includes(q)
      + 1  per visit (popularity boost, capped at 10)
    Filter out score=0. Sort descending by score.

  Tag prefix match is ranked higher than title contains to make tag-driven
  retrieval feel fast — typing "dep" immediately surfaces "deploy" links.

PART 5 — TITLE FETCHING (server-side)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Why server-side: avoids CORS when fetching arbitrary URLs from the browser.

Implementation in server.mjs (or a small helper):
  - Use Node.js built-in fetch (available Node 18+).
  - Set User-Agent to "Mozilla/5.0 …" to avoid bot blocks.
  - Timeout: AbortController + setTimeout(5000).
  - Read up to first 8KB of response body (enough to find <title>).
  - Parse with regex: /<title[^>]*>([^<]+)<\/title>/i
  - Decode HTML entities (at minimum: &amp; &lt; &gt; &quot; &#39;)
  - Return { title: "..." } or { title: "" } on any failure.
  - Never throw — always return a valid JSON response.

PART 8 — EDGE CASES & NOTES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

- Pasting a URL while link mode search box is focused: detect isUrl(pastedText),
  auto-trigger showAddForm(pastedText) instead of filtering.

- Pasting into ask textarea: only trigger auto-save for clean URL pastes
  (entire pasted string is a URL). Mixed text pastes are ignored.

- Duplicate URLs on ask-paste: silently skip the save if the URL already
  exists in _links. No error, no UI.

- Title backfill race: if the link is deleted before the title fetch returns,
  the PATCH is a no-op (check id still exists before applying).

- fetch-title failures: silently fall back to using the raw URL as title.
  The user can always edit it in link mode.

- Links file missing: treat as empty array, create on first save.

- Tag input format: Space or comma commits a tag. Tags are trimmed and
  lowercased before storage. Empty strings and duplicates are silently dropped.

- Tag prefix scoring priority: typing "dep" must surface "deploy" links before
  a link whose title happens to include "dep". Hence tag-prefix score (+40)
  is higher than title-contains (+20).

- '#' search with no text after it: show all links that have at least one tag.
  This lets the user browse by tag visually.

- Tag autocomplete source is derived from _links at query time — no separate
  tag index to maintain. With <1000 links this is fast enough.

- Tag editor focus trap: while the autocomplete dropdown is open, ↑/↓ move
  within the dropdown. Escape closes the dropdown but keeps the tag editor open.

- The overlay should NOT interfere with textarea keypresses in the main app
  because it is hidden when not in link mode. The Ctrl+L handler checks
  event.ctrlKey && event.key === 'l' and calls event.preventDefault().

- Tab order: link mode is a modal overlay; focus should be trapped inside it
  (Tab should cycle between search and status form fields only).

- No Qdrant, no embeddings needed. Links are a simple CRUD store.

- Future idea: "import from browser bookmarks" — accept exported HTML
  bookmark file, parse <DT><A HREF=...> entries, bulk-add. Out of scope now.
