Q: How should we scale to multiple nodes?

Single-node architecture:
  [HTTP server] → [retriever] → [SQLite vectors + docs] → [LLM API]

Multi-node concerns:
  1. Document storage must be shared → NFS mount or MinIO (S3-compatible, self-hosted)
  2. Vector index must be shared → Qdrant cluster (has built-in replication + sharding)
  3. LLM calls are stateless → each node calls the API independently, no coordination needed
  4. Ingest workers can be parallel → multiple nodes ingest different files, write to shared Qdrant
  5. Query nodes → stateless, read from shared Qdrant + doc store, load-balance with nginx

Proposal for multi-node path:
  - Keep all I/O behind module interfaces (storage.mjs, vectors.mjs) switchable via system.toml
  - system.toml: storage.backend = "sqlite" | "postgres" | "s3"
                 vectors.backend = "sqlite" | "qdrant"
  - Nodes are identical; role (ingest / query / both) set in config
  - No central coordinator needed; Qdrant handles vector consistency

This means the codebase itself does not change when scaling — only config does.
